chapter 1 — what is openclaw

What Is OpenClaw, Exactly?

第一章：它到底是个什么东西？

The one-sentence answer: OpenClaw is a self-hosted AI agent gateway that runs on your own computer. When you use ChatGPT, your message travels to OpenAI's servers and back — your conversation runs on someone else's machine. OpenClaw reverses that: the control plane lives on your machine, but you communicate through the chat apps you already have — Telegram, Slack, WhatsApp.

先说最重要的一句话：OpenClaw 是一个跑在你自己电脑上的 AI 助手调度系统。

这话听起来还是有点绕，我们对比一下就明白了。你平时用 ChatGPT，你发一条消息，它跑到 OpenAI 的服务器上兜一圈再回来——你的对话全程走的是别人的机器。OpenClaw 的思路反过来：让这个调度系统住在你自己的电脑或服务器上，但你不需要专门打开一个 App 跟它说话，用你已经装了的 Telegram、Slack、WhatsApp 就行。

In architectural terms, it is a self-hosted, multi-modal AI Agent gateway. Unpacking that:

在架构术语里，这叫 自托管多模态 AI Agent 网关。拆开说：

Self-hosted means the control plane runs on your own machine — no third-party servers in the middle. Multi-modal means it handles text, images, audio, and video. AI Agent means the AI doesn't just reply — it can call tools, observe results, and reason in a loop. Gateway means all traffic flows through one central point.

  • 自托管：跑在你自己机器上，不依赖第三方平台
  • 多模态：能处理文字、图片、音频、视频
  • AI Agent：不只是聊天，还能主动调用工具做事
  • 网关：所有流量的统一入口和出口

The entire project is written in TypeScript, runs on Node.js 22+, and its monorepo is managed with pnpm workspaces.

整个项目用 TypeScript 写的，跑在 Node.js 22+ 上，代码仓库用 pnpm workspaces 管理（这是一种把多个子项目放在同一个代码仓库里的方式，叫 monorepo）。

chapter 2 — the gateway

The Gateway: Your Machine's Always-On Switchboard

第二章：Gateway——整个系统的神经中枢

When you run openclaw gateway --port 18789, a single long-lived process starts up and becomes the control plane of the whole system. Everything — every chat platform, every AI model, every device — communicates through it. It boots at system startup, restarts on crash, and runs silently as a daemon (managed by launchd on macOS, systemd on Linux).

Gateway 是你装好 OpenClaw 之后一直在后台跑的那个进程，它是整套系统的"总调度员"。你可以把它想象成一个永远开着的服务台，所有来自外部的消息都先到它这里，它来决定交给谁处理，处理完了再通过它把回复发出去。

The first thing Gateway does is open a WebSocket server listening on 127.0.0.1:18789 by default. WebSocket differs from HTTP in a fundamental way: HTTP is "you ask, I answer, conversation over" — each request is independent. WebSocket is "we open a phone line and keep talking" — the connection stays open. Gateway uses WebSocket to maintain persistent connections with all clients (your phone app, CLI tool, web console, and the AI itself).

在技术上，Gateway 做的第一件事是在你电脑上开一个 WebSocket 服务器，默认监听本机的 127.0.0.1:18789。

WebSocket 是什么？它是一种网络通信协议，和你平时浏览网页用的 HTTP 最大的区别是：HTTP 是"你问我答，说完拉倒"，每次请求都是独立的；WebSocket 是"我们建立一条电话线路，之后想说什么随时说"，连接一直保持。Gateway 用 WebSocket 和所有客户端（你的手机 App、命令行工具、Web 控制台、AI 助手本体）保持长连接。

The protocol it exposes is JSON-RPC over WebSocket — a permanent bidirectional pipe. Unlike HTTP where each request is its own conversation, Gateway can push events to clients the moment something happens — most importantly, pushing individual words of an AI reply as they're generated.

Gateway 暴露的通信协议叫 JSON-RPC over WebSocket——用 JSON 格式打包的远程过程调用协议。简单说，就是你通过 WebSocket 发一段 JSON，告诉 Gateway 你要调用什么方法、参数是什么，Gateway 处理完给你回一段 JSON。

When a client connects, it sends a handshake first:

连接建立时客户端要先打个招呼：

const ws = new WebSocket('ws://127.0.0.1:18789');

ws.on('open', () => {
  ws.send(JSON.stringify({
    method: 'client.hello',
    params: { type: 'cli', version: '2025.3.10' }
  }));
});

ws.on('message', (data) => {
  const msg = JSON.parse(data);
  // msg.method — event type (what happened)
  // msg.params — event data (the actual content)
  // msg.id     — request ID, used to match requests to replies
});

The Internal Broadcast Station — Event Bus

Gateway 内部的"广播站"——Event Bus

Inside, modules communicate through a Pub/Sub Event Bus. A Channel Adapter publishes an inbound_message event; the Session Manager subscribes, decides routing; the Agent publishes stream.chunk events; the Gateway forwards them over WebSocket. No module needs to know the internals of any other — that's the point of decoupling.

Gateway 内部有一个发布-订阅（Pub/Sub）事件总线，就像广播电台一样：任何模块可以向总线"发布"一个事件，其他任何模块可以"订阅"并监听这个事件。

这个设计的好处是让各模块彼此不需要直接认识：WhatsApp 适配器只管"我收到消息了，向总线广播一下"；Session 管理器只管"监听消息事件，路由给对应对话"；它们之间完全不知道对方存在。这叫做解耦，改一个模块不会牵连其他模块。

How Gateway Auto-starts on Boot

Gateway 怎么做到开机自启？

Gateway is a daemon process, managed by the OS service manager. On macOS it uses launchd with a .plist config file. The key fields are RunAtLoad (start on login) and KeepAlive (restart automatically if it crashes):

Gateway 是一个守护进程（Daemon），由操作系统的服务管理器托管，机器重启后自动拉起。macOS 上使用 launchd，通过一个 .plist 配置文件描述如何启动进程：

<!-- ~/Library/LaunchAgents/ai.openclaw.gateway.plist -->
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>ai.openclaw.gateway</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/local/bin/node</string>
    <string>/path/to/openclaw/dist/gateway.js</string>
    <string>--port</string>
    <string>18789</string>
  </array>
  <key>RunAtLoad</key><true/>   <!-- 登录时自动启动 / Start on login -->
  <key>KeepAlive</key><true/>   <!-- 崩溃后自动重启 / Restart on crash -->
  <key>StandardOutPath</key>
  <string>~/.openclaw/logs/gateway.log</string>
</dict>
</plist>

On Linux, systemd works on the same principle: define the process path, startup arguments, and crash-restart policy.

Linux 上用 systemd，原理完全相同：定义进程路径、启动参数、崩溃重启策略。KeepAlive: true 是关键——它告诉 launchd，如果进程意外挂掉，马上重启。

chapter 3 — boot sequence

The 10-Step Boot Sequence

第三章：装好之后，Gateway 启动的这10步

Gateway startup is strictly ordered — each step depends on the previous one completing cleanly. Running openclaw gateway --port 18789 --verbose initialises 10 subsystems in sequence:

运行 openclaw gateway --port 18789 --verbose 之后，Gateway 不是直接开干的，它按照固定顺序依次初始化10个子系统。这个顺序很重要，因为后面的子系统依赖前面的，乱了就会出错：

第1步：读配置文件
   └─ 找 ~/.openclaw/config.yaml（或环境变量 OPENCLAW_CONFIG 指向的路径）
   └─ 把命令行参数（--port 18789 --verbose）和配置文件合并
   └─ 用 JSON Schema 验证配置格式是否合法

第2步：初始化数据库
   └─ 默认用 SQLite（一个单文件数据库，不需要单独安装）
   └─ 运行"数据库迁移"——检查数据库结构是否是最新版本，不是就升级

第3步：启动 Event Bus（内部消息总线）
   └─ 内存中的发布-订阅系统，承载模块间所有通信

第4步：初始化 Session Manager（会话管理器）
   └─ 从数据库把之前的对话状态恢复到内存
   └─ 注册对总线事件的监听

第5步：初始化 Pi Agent Runtime（AI 助手的运行时）
   └─ 加载所有已安装的技能（Skills），把工具注册进来
   └─ 初始化每个对话用哪个 AI 模型

第6步：同时启动所有 Channel 适配器
   └─ WhatsApp：Baileys 库连接（首次需要扫码，之后自动恢复）
   └─ Telegram：grammY 库开始轮询或 Webhook
   └─ Discord：discord.js 连接 Discord 的 WebSocket
   └─ 其他平台各自独立初始化，互不影响

第7步：启动 WebSocket 服务器
   └─ 监听 127.0.0.1:18789
   └─ 注册所有 RPC 方法的处理函数

第8步：启动 HTTP 服务器
   └─ 提供 Web 控制台（Control UI）
   └─ /webchat 路径提供网页版聊天界面
   └─ /webhook/{token} 路径接受外部触发

第9步：启动 Cron 调度器
   └─ 从数据库恢复之前设定的定时任务
   └─ 把它们注册回 node-cron 调度器

第10步：健康检查端点就绪
    └─ GET /health 返回 {"status":"ok","version":"..."} 表示一切正常
💡 Why order matters为什么顺序很重要 If the Event Bus started before the Session Manager, messages arriving during boot could be published with no subscriber yet — silently dropped. The fixed sequence prevents this entire class of race condition. 如果 Event Bus 在 Session Manager 之前启动，启动期间到达的消息会被发布到总线，但还没有订阅者——消息就会悄悄丢失。固定顺序可以防止这整类竞态条件。

config.yaml is the master configuration file that controls all of this:

config.yaml 是控制所有这些行为的主配置文件：

gateway:
  bind: loopback            # loopback=只有本机能访问 | all=局域网也能访问
  port: 18789
  auth:
    mode: password          # none=不验证 | password=密码 | tailscale=用Tailscale身份
    password: "your-secret"
  tailscale:
    mode: serve             # off=关闭 | serve=内网访问 | funnel=公网访问

workspace:
  name: personal
  model: gpt-4o             # 全局默认模型 / global default model

channels:
  telegram:
    token: "BOT_TOKEN"
    dmPolicy: pairing       # pairing=陌生人需要配对 | open=任何人都能发
    allowFrom: ["123456789"]

models:
  profiles:
    - name: primary
      provider: openai
      model: gpt-4o
      apiKey: "sk-..."
    - name: fallback
      provider: anthropic
      model: claude-3-5-sonnet
      apiKey: "sk-ant-..."
  failover: [primary, fallback]   # 主模型挂了自动切备用 / auto-failover

Why Node.js 22 is required

为什么必须 Node.js 22？

Node 22 is the current LTS (Long-Term Support) release and brings several key features OpenClaw relies on: built-in stable fetch API (no more node-fetch package), native WebSocket client, more stable --experimental-vm-modules (used for sandboxed skill execution), and modern cryptography support in the crypto module.

Node 22 是目前的 LTS（长期支持）版本，带来了几个关键特性：fetch API 正式内置稳定了（之前要装额外的 node-fetch 包）；WebSocket 客户端内置；--experimental-vm-modules 更稳定了（OpenClaw 用这个在沙箱里安全执行技能代码）；crypto 模块支持了更多现代加密算法。

chapter 5 — message lifecycle

A Message's Complete Journey — 2,100 ms End to End

第五章：一条消息的完整旅程——从你发出去到回复出现

Let's trace one message in real time. You send "Check today's weather in Beijing" on Telegram. Here's what happens, millisecond by millisecond:

我们用一个具体例子走完整个流程：你在 Telegram 发了一条消息"帮我查今天北京的天气"。

  • [0 ms] You tap Send in Telegram.
  • [10 ms] Telegram's servers push the event to grammY's long-polling loop. The message handler fires.
  • [12 ms] Channel Adapter converts the raw Telegram object into a typed InboundMessage and publishes it to the Event Bus.
  • [15 ms] Security Layer queries channel_allowlist in SQLite — your ID is there. Allowed.
  • [16 ms] Router resolves the session key: "telegram:123456789". Private message → this Session.
  • [18 ms] Session Manager reads the last 50 messages from SQLite into memory.
  • [20 ms] Gateway calls bot.api.sendChatAction(chatId, 'typing') — you see the typing indicator appear.
  • [25 ms] Pi Agent assembles the full Chat Completions payload: history + new message + tool list + system prompt.
  • [30 ms] HTTP POST to OpenAI with stream: true.
  • [800 ms] First SSE chunk arrives: {"delta":{"content":"Sure"}}. Gateway pushes stream.chunk to all clients.
  • [850 ms] Model emits tool_calls — it wants to call weather_get. Text output stops.
  • [855 ms] Agent executes weather_get({ city: "Beijing", units: "metric" }) via HTTP.
  • [1200 ms]Weather API returns { temp: 22, condition: "Sunny", humidity: 45 }. Appended to history.
  • [1205 ms]Second LLM call begins — now with the tool result as a role: "tool" message.
  • [1800 ms]Final reply streams: "Beijing today — sunny, 22°C, humidity 45%. Good day to go out."
  • [2000 ms]stream.done event emitted. Session Manager writes the full exchange to SQLite.
  • [2015 ms]Channel Adapter sends the reply to Telegram via Bot API.
  • [2100 ms]Your phone displays the reply. Total: 2.1 seconds, one tool call included.

第一站：Telegram 适配器接收消息

OpenClaw 连接 Telegram 用的是 grammY 这个库。Telegram 提供了机器人 API，有两种接收消息的方式：

长轮询（Long Polling）：你的程序主动问 Telegram "有没有新消息？"，Telegram 要么立刻回"有，这条"，要么等最多1秒才回"没有"。不需要公网 IP，但理论上最多有1秒延迟。

Webhook：反过来，Telegram 主动把新消息 POST 到你指定的 HTTPS 地址。实时，延迟几乎为零，但你的服务器必须有公网 IP 和合法 HTTPS 证书。

收到消息后，grammY 把 Telegram 的原始数据格式转换成 OpenClaw 内部统一的消息格式 InboundMessage：

interface InboundMessage {
  id: string;        // Telegram 的消息 ID
  channel: string;   // 'telegram'
  peerId: string;    // 发送者的 Telegram 用户 ID
  groupId?: string;  // 如果是群消息，群组 ID 在这里
  text?: string;     // "帮我查今天北京的天气"
  media?: MediaRef[];// 如果有图片/文件，媒体引用在这里
  replyTo?: string;  // 如果是回复某条消息，原消息 ID 在这里
  timestamp: number; // 消息时间戳（Unix 毫秒）
  raw: unknown;      // Telegram 原始数据，调试用
}

第二站：安全检查——这个人有资格说话吗？

Gateway 做的第一件事不是让 AI 回答，而是查"白名单"。白名单存在 SQLite 里：

CREATE TABLE channel_allowlist (
  channel   TEXT NOT NULL,   -- 'telegram'
  peer_id   TEXT NOT NULL,   -- 对方的用户 ID

第三站：路由——这条消息发给哪个"对话"？

  // 群组消息：每个群有自己独立的 Session
  if (msg.isGroup) {
    return `${msg.channel}:${msg.groupId}`;
  }
  // 私聊：默认路由到 "main" Session
  const agentConfig = findAgentConfig(msg.channel, msg.peerId);
  return agentConfig?.sessionId ?? 'main';
}

第四站：Pi Agent——构建发给 AI 的请求

Agent 从数据库读出这个 Session 的历史消息，加上当前新消息，加上所有可用工具的说明，打包成一个标准的 Chat Completions 请求。注意 "stream": true——这是告诉 AI "别等全部想好了再回复，生成一个字就给我发一个字"。

第五站：ReAct 循环——AI 如何"边想边做"

AI 的流式输出使用 SSE（Server-Sent Events） 格式，每生成一点内容就发一行 data: 打头的数据。Agent 收到每一个小片段（chunk），立刻通过 Gateway WebSocket 转发给你的客户端（stream.chunk 事件）。这就是为什么你在聊天界面看到的是"字一个一个出现"。

第六站：回复发出去

工具执行完，LLM 生成最终文字回复，Agent 把它交回给 Gateway，Gateway 通过 Telegram 适配器发出。如果回复太长，Gateway 会自动分段发送——Telegram 单条消息上限大约 4096 字符，Discord 是 2000 字符，超过就自动切割。

一次含工具调用的完整问答，大约2秒。

Context Window Management

AI 的"记忆上限"——上下文窗口管理

LLMs have hard token limits — GPT-4o tops out at 128K tokens; Claude allows 200K. OpenClaw monitors usage continuously and applies two strategies when you approach the limit:

AI 模型有个硬性限制叫上下文窗口，就是它每次能"看到"的文字总量上限。GPT-4o 是 128K tokens，Claude 是 200K tokens（1个 token 大约等于半个英文单词，或1个汉字）。对话历史越来越长，总有撑满的一天，这时 OpenClaw 有两种处理策略：

Auto-trim silently drops the oldest messages until total usage falls below 80% of the model's window. Fast, but lossy. /compact sends the oldest 70% of history to the LLM, asks for a concise summary, replaces those messages with the summary, and keeps the most recent 30% verbatim. More expensive but preserves far more information:

自动裁剪：悄悄把最早的消息丢掉，保留最近的，让总量低于阈值（比如 80% 的窗口上限）。快但会丢失早期信息。

/compact 摘要压缩：把早期历史发给 AI，让它生成一段摘要，然后用这段摘要替换掉早期的原始消息。信息损失更少，但多花一次 API 调用：

async function compactSession(sessionId: string) {
  const messages  = await db.getMessages(sessionId);
  const keepCount = Math.floor(messages.length * 0.3);
  const toSummarize = messages.slice(0, messages.length - keepCount);
  const toKeep      = messages.slice(-keepCount);

  const summary = await llm.complete({
    messages: [
      { role: 'system', content: '请用简洁的中文总结以下对话的关键信息：' },
      ...toSummarize
    ]
  });

  await db.replaceMessages(sessionId, [
    { role: 'system', content: `[以下是之前对话的摘要]\n${summary}\n[摘要结束]` },
    ...toKeep
  ]);
}

这对应 AI 模型的"推理预算"。OpenAI 的 o1/o3 系列和 Anthropic 的 extended thinking 模式，允许模型在给出答案之前先花一些 token "在脑子里想"：

off      → 不启用推理模式（普通聊天）

high     → 约 20K tokens 的思考

OpenClaw 把这个级别映射到各个模型的实际参数：对 OpenAI 是 reasoning_effort，对 Anthropic 是 budget_tokens。

chapter 8 — the react loop

The ReAct Loop: How the AI Decides to Do Things

第八章：ReAct 循环——AI 如何"边想边做"

OpenClaw's Agent runtime runs a ReAct loop (Reasoning + Acting). Rather than returning one answer, the LLM is allowed to call tools, observe results, and reason further — as many times as needed — before producing a final text reply.

这是整个系统最核心的逻辑，叫做 ReAct Loop（推理-行动循环）。AI 不是一次就能给出完整答案的，它需要反复"想 → 做 → 看结果 → 再想 → 再做"：

Failover: Auto-switching When the Primary Model Goes Down

故障转移：主模型挂了自动换备用

FailoverModelProvider wraps multiple Providers in priority order. On retryable errors (429 rate limit, 503 service unavailable, network timeout), it switches to the next one:

FailoverModelProvider 包装了多个 Provider，按优先级排列。遇到可重试的错误（429 速率限制、503 服务不可用、网络超时），就切换到下一个：

class FailoverModelProvider implements ModelProvider {
  private providers: ModelProvider[];
  private currentIndex = 0;

  async *stream(request: CompletionRequest) {
    let lastError: Error | null = null;

    for (let i = 0; i < this.providers.length; i++) {
      const provider = this.providers[this.currentIndex % this.providers.length];
      try {
        yield* provider.stream(request);
        return;
      } catch (err) {
        lastError = err as Error;
        if (isRetryableError(err)) {
          console.warn(`${provider.name} 失败（${err.status}），切换到下一个…`);
          this.currentIndex++;
          continue;
        }
        throw err;
      }
    }
    throw lastError ?? new Error('所有模型提供商均失败');
  }
}

function isRetryableError(err: unknown): boolean {
  if (err instanceof APIError)
    return [429, 500, 502, 503, 504].includes(err.status);
  return err instanceof NetworkError;
}

class RotatingKeyProvider {
  private keys: string[];
  private index = 0;
  private keyStats = new Map<string, { requests: number; errors: number }>();

  nextKey(): string {
    const key = this.keys[this.index % this.keys.length];
    this.index++;
    return key;
  }

  markError(key: string) {
    const stats = this.keyStats.get(key) ?? { requests: 0, errors: 0 };
    stats.errors++;
    this.keyStats.set(key, stats);
    // 错误率超过50%，临时把这个 Key 从轮询里踢出去
    if (stats.errors / (stats.requests || 1) > 0.5) {
      this.keys = this.keys.filter(k => k !== key);
      setTimeout(() => this.keys.push(key), 60_000);  // 1分钟后回来 / back after 1 minute
    }
  }

Device nodes (phone, computer) connect to the Gateway via WebSocket, register capabilities with node.hello, and get invoked remotely with node.invoke. Skills are defined in YAML, loaded at startup, and extend the AI's capabilities at any time. The model layer abstracts all LLM providers behind a unified interface, with failover and key rotation for high availability.

设备节点（手机、电脑）通过 WebSocket 连到 Gateway，用 node.hello 注册能力清单，Gateway 需要时发 node.invoke 远程调用。技能（Skill）用 YAML 定义工具和提示词，Gateway 启动时加载，让 AI 的能力随时可扩展。模型层用统一接口抽象所有 LLM 提供商，支持故障转移和 Key 轮换，保证可用性。

OpenClaw's design rests on a few explicit principles: Single control plane — one Gateway, one config, one database; no distributed state, no cluster. Interface-driven decoupling — the Gateway never imports WhatsApp directly; it imports a ChannelAdapter. Replacing any component is a local change. Streaming first — every layer from LLM SSE output to WebSocket events is built around streaming. Local-first privacy — your conversation history lives in a SQLite file on your machine. Only the AI API you've explicitly chosen ever sees your messages.

OpenClaw 的设计坚持了几个准则。单一控制平面：没有纷繁复杂的微服务和分布式状态同步，一切数据本地保存，随时可重启和排查。面向接口解耦：网关层不用关心下面跑的是 Telegram 还是手机闹钟，替换组件时不会牵一发而动全身。流式优先：低延迟是最基本的尊严。本地优先的默认隐私护城河：所有数据默认存在你自己硬盘的 SQLite 文件，除了你指定的模型厂商，没有任何第三方看得到你的聊天记录。

The entire data path never touches any third-party server. That feels worth understanding.

整条链路，数据不经过任何第三方服务器。这整套系统值得你的理解。

Appendix A: Try It Yourself (Local Deployment Guide)

Deploying on an Ubuntu VPS (Systemd)

If you prefer hosting OpenClaw on an always-on Ubuntu machine, configure it to run as a systemd service:

# 1. Install Node.js 22 and npm
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt-get install -y nodejs

# 2. Install OpenClaw globally
sudo npm install -g openclaw@latest

# 3. Create a systemd service file (run as your current user, do not run as root)
mkdir -p ~/.config/systemd/user
cat << 'EOF' > ~/.config/systemd/user/openclaw-gateway.service
[Unit]
Description=OpenClaw Gateway
After=network-online.target

[Service]
Type=simple
ExecStart=/usr/bin/env openclaw gateway --port 18789
Restart=on-failure
RestartSec=5s
StandardOutput=append:%h/.openclaw/logs/gateway.log
StandardError=append:%h/.openclaw/logs/gateway.err
MemoryMax=1G

[Install]
WantedBy=default.target
EOF

# 4. Enable and start the service
systemctl --user daemon-reload
systemctl --user enable openclaw-gateway
systemctl --user start openclaw-gateway

# 5. Check status
systemctl --user status openclaw-gateway

You can then use tailscale serve or SSH port forwarding to access your remote Gateway from your local machine.
